# Super Chat: подробная документация на простом языке

## 1. Что это за проект

`super-chat` — это система, которая помогает превратить поток переписки из Telegram в более полезный рабочий формат.

Если объяснять совсем просто, идея такая:

1. Пользователь подключает свой Telegram.
2. Сообщения из Telegram попадают в Matrix через мост `mautrix-telegram`.
3. Наше приложение читает эти сообщения из Matrix.
4. Приложение пытается понять, что в сообщениях важно:
   - задача
   - встреча
   - обещание
   - ожидание ответа
5. Потом оно показывает это в понятном виде:
   - `Today`
   - `Waiting`
   - `Search`

То есть проект не является "ещё одним Telegram-клиентом". Он не пытается полностью заменить Telegram. Он пытается построить поверх переписки рабочий слой:

- что нужно сделать
- о чём нужно помнить
- кого ждём
- что было обещано

## 2. Главная идея системы

Очень важно понимать одну вещь:

`Super Chat` не подключается к Telegram напрямую из самого веб-приложения.

Между Telegram и приложением есть несколько промежуточных слоёв:

`Telegram -> mautrix-telegram -> Synapse (Matrix) -> Super Chat`

Это сделано специально.

Почему так:

- Telegram напрямую неудобно использовать как "внутреннюю шину" приложения.
- Matrix даёт более удобный и стандартный API для чтения комнат и сообщений.
- `mautrix-telegram` уже умеет быть мостом между Telegram и Matrix.
- Нам проще строить продукт поверх Matrix-событий, чем писать весь Telegram ingestion с нуля.

## 3. Большая схема системы

Ниже логическая схема:

1. Пользователь входит в `SuperChat.Web`.
2. Приложение создаёт для него скрытую Matrix-личность.
3. Пользователь запускает подключение Telegram.
4. Приложение через Matrix пишет bridge-боту команду `login`.
5. `mautrix-telegram` связывает Telegram-аккаунт пользователя с его скрытой Matrix-личностью.
6. Telegram-чаты начинают отображаться как Matrix-комнаты.
7. Наш фоновый sync читает эти комнаты.
8. Сообщения сохраняются в `normalized_messages`.
9. Фоновый extractor превращает сообщения в `extracted_items`.
10. UI и API показывают производные данные.

Если кратко:

- Matrix здесь нужен как транспортный слой.
- PostgreSQL нужен как постоянное хранилище продукта.
- Web и API — это два входа в одну и ту же бизнес-логику.

## 4. Из чего состоит проект

Проект состоит из нескольких частей.

### 4.1. `SuperChat.Web`

Это основной веб-интерфейс.

Он отвечает за:

- вход по magic link
- подключение Telegram
- страницы `Today`
- страницы `Waiting`
- поиск
- настройки
- отправку feedback

Это обычный ASP.NET Core Razor Pages хост.

Файл входа:

- `src/SuperChat.Web/Program.cs`

### 4.2. `SuperChat.Api`

Это отдельный JSON API для будущих мобильных клиентов, интеграций и машинного доступа.

Он отвечает за:

- выдачу API-сессий
- `GET /api/v1/me`
- `GET /api/v1/work-items/today`
- `GET /api/v1/work-items/waiting`
- `GET /api/v1/search`
- работу с Telegram integration через API

Важно:

API не запускает фоновые воркеры синка и extraction.

Это специально сделано, чтобы не было дублирования фоновых задач сразу из двух процессов.

Файл входа:

- `src/SuperChat.Api/Program.cs`

### 4.3. `SuperChat.Contracts`

Это слой общих контрактов.

Здесь лежат:

- option-классы конфигурации
- общие view model
- классы, которые должны быть понятны сразу нескольким слоям

Примеры:

- `PilotOptions`
- `MatrixOptions`
- `TelegramBridgeOptions`
- `DeepSeekOptions`
- `EmailOptions`
- `PersistenceOptions`

Смысл этого слоя:

если настройка или контракт нужны и Web, и API, и Infrastructure, лучше держать их в одном месте.

### 4.4. `SuperChat.Domain`

Это слой доменной логики.

Здесь лежат:

- доменные модели
- enum'ы
- композиция итоговых данных

Например:

- `ExtractedItemKind`
- `DigestComposer`

Смысл слоя:

здесь не должно быть HTTP, SQL, Matrix API или Docker. Это слой правил и смыслов.

### 4.5. `SuperChat.Infrastructure`

Это самый "рабочий" слой.

Именно здесь находится большая часть реальной логики:

- доступ к базе
- Matrix API клиент
- provisioning Matrix-пользователя
- запуск Telegram login flow
- sync сообщений
- нормализация сообщений
- extraction
- digest
- search
- feedback
- health snapshot
- фоновые воркеры

Если коротко:

`Infrastructure` — это слой, который реально общается с внешним миром.

## 5. Как устроены зависимости между слоями

Зависимости в проекте идут сверху вниз:

- `Web` зависит от `Infrastructure`, `Domain`, `Contracts`
- `Api` зависит от `Infrastructure`, `Domain`, `Contracts`
- `Infrastructure` зависит от `Domain` и `Contracts`
- `Domain` ни от чего прикладного не зависит
- `Contracts` ни от чего прикладного не зависит

Почему это важно:

- так проще менять UI, не ломая базовую логику
- так проще писать тесты
- так проще понимать, где именно должна жить новая функциональность

Например:

- новую кнопку надо делать в `Web`
- новую интеграцию с Matrix надо делать в `Infrastructure`
- новую настройку надо добавлять в `Contracts`
- новые правила формирования `Today` надо делать в `Domain`

## 6. Какие внешние системы используются

Проект интегрируется с несколькими внешними системами.

### 6.1. PostgreSQL

PostgreSQL — это основная база данных приложения.

В ней хранятся:

- приглашённые email'ы
- пользователи
- magic links
- API-сессии
- Matrix identity пользователя
- состояние Telegram-подключения
- checkpoint синка
- нормализованные сообщения
- extracted items
- feedback

Это главная база продукта.

Отдельно в той же инфраструктуре есть ещё базы для:

- `synapse`
- `mautrix_telegram`

То есть всего баз несколько, но для логики продукта ключевая — `superchat_app`.

### 6.2. Synapse

Synapse — это Matrix homeserver.

Он нужен как центральный Matrix-сервер.

Что он делает в этом проекте:

- хранит Matrix-пользователей
- хранит Matrix-комнаты
- выдаёт Matrix access token
- принимает команды от нашего приложения
- отдаёт события через Matrix Client API

Наше приложение общается с Synapse через `MatrixApiClient`.

### 6.3. mautrix-telegram

`mautrix-telegram` — это мост между Telegram и Matrix.

Его задача:

- взять Telegram-аккаунт пользователя
- представить его внутри Matrix
- превратить Telegram-чаты в Matrix-комнаты
- превратить Telegram-сообщения в Matrix-события

Очень важно:

наш продукт не парсит Telegram-сообщения напрямую из Telegram API в основном приложении.

Он получает их уже после прохождения через bridge.

### 6.4. mautrix-telegram-helper

Это маленький вспомогательный сервис, который мы добавили рядом с bridge.

Зачем он нужен:

- bridge знает настоящие данные Telegram-комнаты
- но приложению нужно быстро и просто понимать, что это за комната
- особенно важны:
  - размер группы
  - является ли комната broadcast-каналом

Почему helper вообще понадобился:

Matrix membership не показывает правду о размере Telegram-группы.

Например:

- в Telegram группа может иметь 356 участников
- а в Matrix portal room реально "видимых" участников может быть только 10

Если бы мы смотрели только на Matrix, мы бы ошибались.

Поэтому helper использует данные bridge и Telethon-сессию, чтобы вернуть:

- `peer_type`
- `participant_count`
- `title`
- `is_broadcast_channel`

Именно это теперь используется в фильтре sync.

### 6.5. Caddy

`Caddy` — это внешний веб-сервер и reverse proxy.

Он нужен для:

- TLS / HTTPS
- маршрутизации доменов
- проксирования:
  - `app.*` -> `superchat-web`
  - `api.*` -> `superchat-api`
  - `matrix.*` -> `synapse`
  - `bridge.*` -> `mautrix-telegram`

### 6.6. DeepSeek

DeepSeek задуман как LLM-провайдер для extraction.

Важно понимать текущее состояние:

- конфигурация под DeepSeek уже есть
- seam под реальный AI вызов тоже есть
- но текущая реализация всё ещё bootstrap
- фактически сейчас extraction опирается на эвристику

То есть реального production-grade LLM extraction пока нет.

Это важно, потому что пользователь может ожидать "настоящий AI", а пока поведение ближе к умной эвристике.

### 6.7. Mailpit

`Mailpit` используется локально для разработки.

Он нужен, чтобы удобно смотреть "отправленные письма" с magic link без настоящего SMTP-сервера.

В production вместо него используется обычный SMTP.

## 7. Как работает вход пользователя

### 7.1. Приглашение

Проект работает по модели invite-only.

Это значит:

- просто любой email войти не может
- email должен быть заранее разрешён в `pilot_invites`

### 7.2. Magic link

Когда пользователь вводит email:

1. Приложение нормализует email.
2. Проверяет, есть ли активное приглашение.
3. Создаёт одноразовый токен.
4. Сохраняет его в таблицу `magic_links`.
5. Формирует ссылку вида:
   - `/auth/verify?token=...`

Это делает `AuthFlowService`.

### 7.3. Проверка токена

Когда пользователь нажимает ссылку:

1. Токен ищется в БД.
2. Проверяется, что он не просрочен.
3. Если пользователя ещё нет, создаётся `app_users`.
4. Токен помечается как использованный.
5. После этого создаётся или подтверждается Matrix identity пользователя.

То есть успешный логин в приложение сразу тянет за собой Matrix-подготовку.

## 8. Как создаётся скрытая Matrix-личность

Для каждого пользователя продукта существует скрытый Matrix-пользователь.

Это делает `MatrixProvisioningService`.

Логика такая:

1. Приложение строит Matrix user id по email.
   - обычно это что-то вроде `@superchat-username:matrix.example.com`
2. Если есть `Matrix__AdminAccessToken`, приложение через Synapse Admin API:
   - создаёт Matrix-пользователя
   - логинится от его имени
   - получает access token
3. Если admin token не настроен, используется bootstrap-режим:
   - создаётся dev-token

Зачем нужна скрытая Matrix-личность:

- именно от её имени приложение общается с bridge
- именно её комнаты и события потом синкаются
- это "технический пользователь", а не UI-аккаунт человека в самом приложении

## 9. Как запускается подключение Telegram

Это делает `TelegramConnectionService`.

Логика подключения такая:

1. Пользователь нажимает "подключить Telegram".
2. Приложение убеждается, что Matrix identity уже есть.
3. Если для пользователя ещё нет management room:
   - создаётся приватная Matrix-комната
   - туда приглашается bridge-бот
4. В эту комнату отправляется сообщение `login`.
5. Состояние подключения становится `BridgePending`.

Дальше bridge отвечает уже со своей стороны.

### Что такое management room

Это специальная техническая комната между:

- скрытым Matrix-пользователем
- bridge-ботом

Она нужна не для обычных сообщений.

Она нужна для служебных команд:

- `login`
- `logout`
- технические ответы bridge

Очень важно:

эта комната не должна попадать в продуктовые данные.

Именно поэтому при sync она отдельно отфильтровывается.

## 10. Как реальные сообщения доходят до приложения

Это самый важный технический поток во всей системе.

### Шаг 1. Telegram

Пользователь пишет или получает сообщения в Telegram.

### Шаг 2. Bridge

`mautrix-telegram` получает эти события из Telegram и создаёт соответствующие Matrix portal rooms.

### Шаг 3. Synapse

Эти portal rooms и события живут уже в Matrix.

### Шаг 4. Наш sync

`MatrixSyncBackgroundService` периодически делает `/sync` через Matrix API.

Он получает:

- новые комнаты
- новые invite
- timeline events

### Шаг 5. Фильтрация комнат

Перед сохранением приложение решает, нужно ли вообще брать эту комнату.

Сейчас логика такая:

- management room — не брать
- direct chat — брать
- broadcast channel — не брать
- группа с размером больше лимита — не брать
- группа с размером меньше или равным лимиту — брать

По умолчанию лимит:

- `30` участников

Почему так:

- личные переписки полезны почти всегда
- маленькие группы часто рабочие и содержательные
- большие группы и каналы чаще создают шум

### Шаг 6. Сохранение raw-сообщения в продуктовую БД

Если комната разрешена:

1. каждое сообщение нормализуется
2. сохраняется в `normalized_messages`
3. dedup делается по:
   - `user_id`
   - `matrix_room_id`
   - `matrix_event_id`

Это делает `MessageNormalizationService`.

То есть одно и то же сообщение два раза не должно появиться.

## 11. Какие сообщения мы специально не берём

Сейчас есть несколько важных фильтров.

### 11.1. Management room

Мы не берём:

- `login`
- `logout`
- служебные ответы bridge

Иначе они бы попадали в `Today` как будто это реальные рабочие задачи.

### 11.2. Большие группы

Если Telegram-группа слишком большая, мы её игнорируем.

Сейчас правило:

- брать только комнаты с числом участников `<= 30`

### 11.3. Broadcast channels

Telegram-каналы теперь тоже игнорируются.

Почему:

- они обычно не содержат двусторонней рабочей коммуникации
- они создают шум
- нам не нужны новости и широковещательные ленты для личного productivity-слоя

Очень важно:

раньше часть каналов могла уже успеть попасть в БД. Сейчас логика исправлена, и старый хвост был почищен.

## 12. Как приложение понимает, что комната — это канал

Это тонкий момент.

Нельзя просто смотреть:

- `peer_type == channel`

Потому что в Telegram и мегагруппа, и broadcast-канал могут идти как `channel`.

Поэтому мы дополнительно определяем:

- это broadcast channel или нет

Логика определения делается в `mautrix-telegram-helper`:

1. helper читает room mapping из БД bridge
2. достаёт Telegram entity через Telethon
3. смотрит признаки канала
4. возвращает `is_broadcast_channel`

В результате:

- большая мегагруппа может быть `channel`, но `is_broadcast_channel = false`
- настоящий канал будет `channel` и `is_broadcast_channel = true`

Это важно, потому что:

- рабочие группы нам нужны
- каналы нам не нужны

## 13. Что такое normalized message

`normalized_messages` — это не "сырой dump Matrix JSON".

Это уже приведённое к простому формату сообщение:

- кто написал
- что написал
- когда написал
- из какой комнаты
- к какому Matrix event относится

Это промежуточный слой между транспортом и AI/эвристикой.

Зачем он нужен:

- удобно дедуплицировать
- удобно повторно обрабатывать
- удобно отделить ingest от extraction

## 14. Как работает extraction

Extraction — это этап, на котором приложение пытается понять смысл сообщения.

Например:

- "please send the proposal tomorrow" -> задача
- "we have a call Friday at 11" -> встреча
- "I'll send the contract by end of day" -> обещание
- "still waiting for reply from Marina" -> ожидание ответа

### Как это работает сейчас

Есть `ExtractionBackgroundService`.

Он:

1. каждые несколько секунд берёт непросмотренные `normalized_messages`
2. передаёт их в `IAiStructuredExtractionService`
3. получает список `ExtractedItem`
4. сохраняет их в `extracted_items`
5. помечает исходные сообщения как `processed = true`

### Какие бывают типы карточек

Сейчас есть 4 типа:

- `Task`
- `Meeting`
- `Commitment`
- `WaitingOn`

### Что реально используется сейчас

Хотя интерфейс сделан под AI, текущая практическая логика остаётся bootstrap:

- если DeepSeek не настроен, используется эвристика
- если DeepSeek настроен, seam уже есть, но текущая реализация всё равно фактически использует эвристику

То есть это не "большая LLM-система", а пока аккуратный переходный этап.

### Как работает эвристика

`HeuristicStructuredExtractionService` ищет в тексте характерные фразы:

- для встреч
- для задач
- для обещаний
- для ожиданий ответа

Если ничего не найдено, делается fallback:

- `Task`
- заголовок `Follow-up candidate`

Из-за этого иногда шумные сообщения могут превращаться в слишком общие карточки.

Это текущее ограничение системы.

## 15. Что такое extracted item

`extracted_items` — это уже не сообщение, а смысловая карточка.

У неё есть:

- тип
- заголовок
- summary
- комната-источник
- event-источник
- человек
- observedAt
- dueAt
- confidence

Это уже слой продукта, а не транспорта.

## 16. Как работает Today

Очень важный момент:

`Today` показывает не сообщения.

`Today` показывает extracted items.

То есть пользователь видит не "чат как чат", а список важных сущностей, которые система извлекла из переписки.

### Как строится Today

1. Берутся все extracted items пользователя.
2. Оставляются только элементы за текущий день.
3. День считается не по UTC, а по бизнес-таймзоне.
4. Сейчас по умолчанию это:
   - `Europe/Moscow`
5. Из них берутся:
   - задачи
   - обещания
   - встречи, если они скоро

Это делает `DigestService` + `DigestComposer`.

### Что Today не делает

`Today`:

- не показывает raw чат целиком
- не показывает всё подряд из БД
- не является Telegram-клиентом

## 17. Как работает Waiting

`Waiting` — это отдельный экран для того, что мы "ждём от кого-то".

Он строится из extracted items типа:

- `WaitingOn`

Это полезно для тем:

- кому мы написали, но не получили ответ
- кто что-то должен был прислать
- где завис вопрос

## 18. Как работает Search

Search работает в два шага.

### Сначала ищем по extracted items

Это лучший случай.

Если система уже поняла смысл сообщения, поиск найдёт:

- title
- summary
- room name

### Если structured-результатов нет

Тогда поиск падает обратно на недавние `normalized_messages`.

Это важно, потому что:

- extractor может что-то не понять
- но пользователь всё равно должен иметь шанс найти текст сообщения

Поэтому Search — это "сначала по смыслу, потом по тексту".

## 19. Как подставляются имена комнат

В БД и во внутренних процессах мы часто видим Matrix room id:

- что-то вроде `!abc123:matrix.example.com`

Для человека это неудобно.

Поэтому есть `MatrixRoomDisplayNameService`.

Он:

1. получает список room id
2. смотрит кэш
3. если нужно, спрашивает Matrix state
4. ищет:
   - `m.room.name`
   - или canonical alias
5. подставляет это имя в UI

Именно поэтому `Today`, `Waiting` и `Search` теперь показывают названия комнат, а не технические id.

## 22. Почему фоновые воркеры живут только в Web

Это архитектурное решение.

У нас есть два хоста:

- `Web`
- `Api`

Если оба одновременно будут крутить sync и extraction, появятся проблемы:

- гонки
- дублирование работы
- лишняя нагрузка
- конфликты в join или sync

Поэтому сейчас:

- `Web` запускает фоновые воркеры
- `Api` их не запускает

Это зашито через:

- `AddSuperChatBootstrap(..., enableBackgroundWorkers: false)` в API

## 26. Что важно понимать про ограничения текущей реализации

Ниже список текущих границ системы.

### 26.2. AI extraction пока упрощённый

Сейчас система специально не берёт:

- management room
- большие группы
- Telegram channels

## 28. Краткий жизненный цикл одного сообщения

Вот самый полезный "короткий рассказ" о том, как одно сообщение проходит систему.

Представим, что человек написал в Telegram:

`Please send the proposal tomorrow`

Что происходит дальше:

1. Сообщение приходит в Telegram.
2. `mautrix-telegram` превращает его в Matrix event.
3. `MatrixSyncBackgroundService` получает это событие через `/sync`.
4. Перед сохранением приложение проверяет:
   - не management room ли это
   - не channel ли это
   - не слишком ли большая группа
5. Если всё хорошо, сообщение пишется в `normalized_messages`.
6. `ExtractionBackgroundService` забирает это сообщение.
7. Эвристика/AI понимает, что это задача.
8. В `extracted_items` появляется карточка `Task`.
9. `Today` показывает пользователю эту карточку.

То есть от обычного сообщения до карточки в интерфейсе проходит несколько этапов, и каждый этап решает свою задачу.

## 29. Самое важное в одной короткой выжимке

Если объяснить проект совсем просто, то:

- Web и API — это два входа в один и тот же продукт.
- Продукт хранит своё состояние в PostgreSQL.
- Matrix нужен как промежуточный слой между Telegram и приложением.
- `mautrix-telegram` превращает Telegram в Matrix.
- Наш sync читает Matrix, а не Telegram напрямую.
- Из сообщений строятся нормализованные записи.
- Из них строятся смысловые карточки.
- `Today`, `Waiting` и `Search` работают уже поверх этих карточек и сообщений.
- Мы специально режем шум:
  - технические комнаты
  - большие группы
  - каналы
