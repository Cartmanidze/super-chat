# Super Chat: как устроен проект изнутри

## Для кого этот документ

Этот документ написан для разработчика, который **впервые видит проект super-chat** и хочет понять, как всё работает. Здесь нет сухих определений — только объяснения простым языком с аналогиями из реальной жизни.

---

## Часть 1. Что это за проект и зачем он нужен

### Проблема

Представьте, что вы каждый день получаете 200+ сообщений в Telegram. Среди них:
- кто-то просил прислать отчёт
- кто-то назначил встречу
- кто-то обещал что-то сделать
- кто-то ждёт от вас ответа

Всё это **тонет** в потоке сообщений. Вы забываете, путаете, теряете.

### Решение

Super Chat — это **умный секретарь**, который:
1. Читает ваш Telegram (с вашего разрешения)
2. Понимает, что в сообщениях важно
3. Показывает это в удобном виде: задачи, встречи, ожидания

**Аналогия:** представьте, что у вас есть ассистент, который сидит рядом, читает вашу переписку, и каждое утро кладёт на стол листок: "Вот что вам нужно сделать сегодня, вот кого вы ждёте, вот какие встречи запланированы". Super Chat делает то же самое, только автоматически.

## Часть 2. Как сообщения попадают из Telegram в приложение

### Почему не напрямую

Super Chat **не подключается к Telegram напрямую**. Между Telegram и приложением стоят два посредника.

**Аналогия с почтой:**
- Telegram — это почтовый ящик отправителя
- `mautrix-telegram` (bridge) — это почтальон, который забирает письма из одного ящика и кладёт в другой
- Synapse (Matrix) — это наш внутренний почтовый ящик
- Super Chat — это сотрудник, который достаёт письма из нашего ящика и обрабатывает

### Цепочка доставки сообщения

```
Telegram        →  mautrix-telegram  →  Synapse (Matrix)  →  Super Chat
(исходник)         (мост/почтальон)      (наш ящик)           (обработчик)
```

### Зачем такая сложная цепочка?

1. **Matrix даёт удобный API.** Вместо того чтобы разбираться с MTProto (протокол Telegram), мы используем простой REST API Matrix.
2. **Bridge уже написан.** `mautrix-telegram` — это готовый open-source проект, который умеет связывать Telegram и Matrix.
3. **Расширяемость.** Завтра мы можем добавить WhatsApp bridge, Signal bridge — и та же логика будет работать.

### Что такое Matrix простыми словами

Matrix — это **протокол для обмена сообщениями**, как email, но для мессенджеров. У него есть:
- **Комнаты** — аналог чатов/групп
- **События** — аналог сообщений
- **Пользователи** — аналог аккаунтов
- **Sync API** — способ получить "что нового произошло с последнего раза"

Synapse — это **сервер**, который реализует протокол Matrix. Как Gmail реализует протокол email.

### Что такое Bridge

**Аналогия:** мост через реку. С одной стороны — Telegram, с другой — Matrix. Bridge стоит посередине и "перебрасывает" сообщения с одного берега на другой.

Когда вы подключаете Telegram:
1. Bridge создаёт **portal rooms** — Matrix-комнаты, которые зеркалят ваши Telegram-чаты
2. Каждое новое сообщение в Telegram автоматически появляется в соответствующей Matrix-комнате
3. Наше приложение видит эти комнаты и сообщения через Matrix API

## Часть 3. Архитектура приложения (слои)

### Аналогия: ресторан

Представьте ресторан:
- **Зал (Web/Api)** — куда приходят гости и делают заказы
- **Кухня (Infrastructure)** — где готовят еду, работают с продуктами, ходят на склад
- **Рецептурная книга (Domain)** — правила: что с чем сочетается, сколько класть соли
- **Меню и прайс-лист (Contracts)** — общие документы, которые понимают и зал, и кухня

### Слои проекта

#### SuperChat.Contracts — "Меню"
Общие договорённости между всеми частями системы:
- Классы конфигурации (`PilotOptions`, `MatrixOptions`, `DeepSeekOptions`)
- Модели запросов и ответов (request/response)
- View models для UI

**Правило:** здесь нет никакой логики. Только структуры данных.

#### SuperChat.Domain — "Рецептурная книга"
Чистая бизнес-логика без привязки к базам, API, сети:
- Доменные сущности (`AppUser`, `ExtractedItem`, `WorkItemRecord`)
- Перечисления (`ExtractedItemKind`, `IntegrationProvider`)
- Правила композиции данных (`DigestComposer`)

**Правило:** этот слой не знает ни про HTTP, ни про SQL, ни про Matrix. Только правила и смыслы.

#### SuperChat.Infrastructure — "Кухня"
Самый большой слой. Здесь живёт вся "рабочая" логика:
- Работа с базой данных (EF Core, `SuperChatDbContext`)
- Matrix API клиент (`MatrixApiClient`)
- Telegram connection flow
- Синхронизация сообщений (background workers)
- AI extraction и resolution
- Поиск, дайджесты, чат
- DI-конфигурация (`ServiceCollectionExtensions`)

**Правило:** если что-то требует обращения к внешнему миру (БД, API, файловая система), оно живёт здесь.

#### SuperChat.Web — "Зал (для людей)"
Web-интерфейс на Razor Pages:
- Страницы входа и magic link
- Подключение Telegram
- Экраны Today, Waiting, Search
- Настройки, обратная связь
- **Запускает background workers** (sync, extraction, chunking)

#### SuperChat.Api — "Зал (для машин)"
JSON API для мобильных клиентов и интеграций:
- Все эндпоинты под `/api/v1/*`
- Bearer-token аутентификация
- **НЕ запускает background workers** (чтобы не было дублирования)

### Почему два "зала"?

**Аналогия:** представьте ресторан с обычным залом и окном для выдачи заказов. В зале сидят люди (Web — для браузера), а в окно подъезжают курьеры (Api — для мобильного приложения). Кухня одна и та же, но входы разные.

**Важно:** фоновые задачи (синк сообщений, extraction) запускаются только в Web. Если бы оба хоста их запускали, было бы как два повара, одновременно солящих один и тот же суп.

## Часть 4. Как работает вход пользователя

### Magic Link — "ключ без замочной скважины"

Вместо пароля используется **magic link** — одноразовая ссылка на email.

**Аналогия:** представьте, что вместо ключа от квартиры вам каждый раз на почту присылают одноразовый QR-код. Вы его сканируете — дверь открывается. Код больше не работает.

### Пошагово

```
1. Пользователь вводит email
   ↓
2. Система проверяет: есть ли этот email в списке приглашённых? (pilot_invites)
   ↓  (если нет — отказ)
3. Создаётся одноразовый токен, действительный 15 минут
   ↓
4. Формируется ссылка: /auth/verify?token=abc123...
   ↓
5. В dev-режиме ссылка возвращается прямо в ответе
   В production — отправляется на email через SMTP
   ↓
6. Пользователь нажимает ссылку → токен проверяется → создаётся сессия
   ↓
7. Автоматически создаётся скрытая Matrix-личность (see below)
```

### Скрытая Matrix-личность

При успешном входе для пользователя **автоматически создаётся Matrix-аккаунт**. Пользователь об этом не знает.

**Аналогия:** когда вы регистрируетесь в банке, банк автоматически открывает вам внутренний расчётный счёт. Вы им не пользуетесь напрямую — но он нужен для работы системы.

**Зачем:** именно от имени этого Matrix-пользователя приложение:
- Создаёт management room с bridge-ботом
- Получает sync-события с новыми сообщениями
- Видит portal rooms с Telegram-чатами

### API-сессии (для мобильных клиентов)

API использует **Bearer token** вместо cookies. Процесс:
```
POST /api/v1/auth/magic-links   → получаем magic link
POST /api/v1/auth/token-exchange → обмениваем token на Bearer session
```

Сессия действует 30 дней. Можно обновить через `/auth/refresh` или завершить через `/auth/logout`.

## Часть 5. Как подключается Telegram

### Общая идея

Пользователь нажимает "Подключить Telegram" → приложение организует связь между его Telegram-аккаунтом и скрытой Matrix-личностью через bridge.

### Пошагово

```
1. Пользователь нажимает "Подключить Telegram"
   ↓
2. Приложение убеждается, что Matrix identity уже есть
   ↓
3. Создаётся management room — приватная Matrix-комната
   ↓
4. В комнату приглашается bridge-бот (@telegrambot:...)
   ↓
5. Ждём, пока бот примет приглашение (до 5 секунд)
   ↓
6. Отправляем в комнату команду "login"
   ↓
7. Bridge отвечает ссылкой для входа в Telegram (web login URL)
   ↓
8. Пользователь открывает ссылку, вводит телефон и код
   ↓
9. Bridge связывает Telegram-аккаунт с Matrix-пользователем
   ↓
10. Telegram-чаты начинают появляться как Matrix portal rooms
```

### Что такое Management Room

**Аналогия:** это как "служебный чат с техподдержкой". Туда отправляются только команды: "login", "logout", "status". Обычные сообщения пользователя туда не попадают.

**Важно:** management room специально **исключается из синхронизации**, чтобы служебные команды не попали в Today/Waiting как "рабочие задачи".

### Состояния подключения

```
NotStarted → BridgePending → Connected → Disconnected
                                ↑
                           RequiresSetup
                                ↑
                              Error
```

## Часть 6. Путь сообщения через систему (Message Pipeline)

Это **самая важная часть** для понимания системы. Давайте проследим путь одного сообщения от начала до конца.

### Пример

Кто-то написал вам в Telegram: *"Пришли отчёт завтра до 15:00"*

### Этап 1: Telegram → Matrix (bridge)

`mautrix-telegram` видит это сообщение и создаёт Matrix-событие в portal room, соответствующей этому чату.

### Этап 2: Matrix → normalized_messages (sync)

`MatrixSyncBackgroundService` — фоновый воркер, который каждые несколько секунд делает запрос `/sync` к Synapse.

**Аналогия:** это как секретарь, который каждые 5 секунд проверяет почтовый ящик: "Пришло что-нибудь новое?"

Перед сохранением срабатывает **фильтр**:
- Management room? → пропускаем
- Broadcast-канал? → пропускаем
- Группа больше 30 человек? → пропускаем
- Личный чат или маленькая группа? → берём!

Если фильтр пройден, сообщение **нормализуется** и сохраняется в таблицу `normalized_messages`:
```
id | user_id | source | matrix_room_id | sender_name | text | sent_at
```

**Дедупликация:** каждое сообщение уникально по паре `(matrix_room_id, matrix_event_id)`. Одно сообщение дважды не попадёт.

### Этап 3: normalized_messages → chunks (chunking)

`ChunkBuilderService` берёт нормализованные сообщения и группирует их в **chunks** (порции).

**Аналогия:** представьте, что вы не читаете письма по одному, а собираете пачки по 5-8 писем из одного разговора. Так проще понять контекст.

Правила:
- Максимум 8 сообщений в chunk
- Максимум 1600 символов
- Если между сообщениями прошло больше 15 минут — это новый chunk

### Этап 4: chunks → embeddings → Qdrant (indexing)

`ChunkIndexingService` берёт готовые chunks и:
1. Отправляет текст chunk'а в **embedding service** (Python-сервис с моделью BGE-M3)
2. Получает вектор (массив из 1024 чисел)
3. Сохраняет вектор в **Qdrant** (векторная БД)

**Аналогия:** embedding — это как "координаты смысла". Два похожих по смыслу текста получат близкие координаты. Потом, когда пользователь задаёт вопрос, мы ищем chunks с ближайшими координатами — то есть самые релевантные по смыслу.

### Этап 5: normalized_messages → extracted_items (extraction)

Параллельно с chunking работает **extraction**. `BootstrapStructuredExtractionService` анализирует сообщения и пытается извлечь структурированные сущности:

| Тип | Что это | Пример |
|-----|---------|--------|
| **Task** | Задача к выполнению | "Пришли отчёт завтра" |
| **Meeting** | Встреча/звонок | "Созвон в пятницу в 11" |
| **Commitment** | Обещание | "Я пришлю договор до конца дня" |
| **WaitingOn** | Ожидание от кого-то | "Жду ответа от Марины" |

Текущая реализация:
- Если DeepSeek настроен → AI extraction
- Если нет → эвристика (поиск характерных фраз)

### Этап 6: extracted_items → work_items + meetings (work item pipeline)

Извлечённые элементы проходят через **work item pipeline**:

1. **Ingestion:** `ExtractedItem` → `WorkItem` или `Meeting`
2. **Auto-resolution:** AI проверяет, не выполнен ли элемент уже (на основе новых сообщений в чате)
3. **Projection:** meetings строятся из chunks и извлечённых данных

Work items бывают трёх типов:
- **Request** — запрос на действие (ожидает ответа)
- **Event** — встреча/событие с датой
- **ActionItem** — задача к выполнению

### Этап 7: Пользователь видит результат

**Today** — все активные work items за сегодня (по таймзоне `Europe/Moscow`)

**Waiting** — элементы типа WaitingOn ("от кого ждём")

**Search** — гибридный поиск:
1. Сначала по work items (структурированные данные)
2. Потом по Qdrant (семантический поиск по chunks)

**Chat (Ask)** — пользователь задаёт вопрос, система:
1. Ищет релевантные chunks в Qdrant
2. Отправляет контекст + вопрос в DeepSeek
3. Возвращает AI-ответ с цитатами

### Полная схема пайплайна

```
Telegram
  ↓ (bridge)
Matrix portal rooms
  ↓ (MatrixSyncBackgroundService)
normalized_messages
  ↓                    ↓
chunks              extraction
  ↓                    ↓
embeddings     extracted_items
  ↓                    ↓
Qdrant          work_items + meetings
  ↓                    ↓
  └──── Search ────────┘
  └──── Chat/Ask ──────┘
  └──── Today ─────────┘
  └──── Waiting ───────┘
```

## Часть 7. Фильтрация комнат — "что берём, а что нет"

### Зачем фильтровать

Без фильтра система бы захлебнулась:
- Каналы новостей создают тонны шума
- Группы на 1000 человек генерируют слишком много нерелевантных сообщений
- Технические комнаты bridge содержат служебные команды

### Как определяем размер группы

**Проблема:** Matrix не показывает реальный размер Telegram-группы. В Telegram может быть 356 участников, а Matrix видит только 10.

**Решение:** `mautrix-telegram-helper` — маленький Python-сервис, который смотрит данные прямо в БД bridge и через Telethon (Telegram API) и возвращает:
- `participant_count` — реальное количество участников
- `is_broadcast_channel` — это канал (односторонняя рассылка)?
- `peer_type` — тип: user, chat, channel
- `title` — название

### Правила фильтрации

| Тип комнаты | Берём? | Почему |
|-------------|--------|--------|
| Личный чат (DM) | Да | Почти всегда содержит полезную информацию |
| Маленькая группа (<=30) | Да | Рабочие группы обычно содержательные |
| Большая группа (>30) | Нет | Слишком много шума |
| Broadcast-канал | Нет | Одностороння рассылка, не рабочая коммуникация |
| Management room | Нет | Служебные команды bridge |

## Часть 8. AI-обработка — Extraction, Resolution, Retrieval, Chat

### Extraction (извлечение)

**Аналогия:** представьте, что вы читаете газету и маркером выделяете ключевые факты: "это задача", "это встреча", "это обещание".

Текущие реализации:
- **HeuristicStructuredExtractionService** — поиск по ключевым фразам (fallback)
- **DeepSeekStructuredExtractionService** — LLM-based extraction (основная)
- **BootstrapStructuredExtractionService** — выбирает между ними

### Resolution (определение статуса)

**Аналогия:** через несколько дней вы проверяете, не решился ли вопрос сам собой. Может, человек уже прислал отчёт? Тогда задачу можно закрыть.

`DeepSeekResolutionService` анализирует новые сообщения в чате и определяет, не был ли work item уже выполнен, отменён или отложен.

### Retrieval (семантический поиск)

**Аналогия:** Google для вашей переписки. Вы спрашиваете "что мне говорили про договор?", а система находит все релевантные фрагменты, даже если слово "договор" не упоминалось напрямую (может быть "контракт" или "соглашение").

Как работает:
1. Вопрос пользователя → embedding (вектор)
2. Поиск ближайших векторов в Qdrant
3. Возврат top-N chunks с текстом и метаданными

### Chat (ответы на вопросы)

**Аналогия:** виртуальный ассистент. Вы спрашиваете: "Какие встречи у меня завтра?", и он отвечает, опираясь на вашу переписку.

Шаблоны вопросов:
| Template | Что делает |
|----------|-----------|
| `today` | Показывает задачи и события на сегодня |
| `waiting` | Показывает, от кого ждём ответа |
| `meetings` | Показывает предстоящие встречи |
| `recent` | Показывает последние события |
| `custom` | Произвольный вопрос пользователя |

Для custom-вопросов система:
1. Фильтрует стоп-слова (мат, спам)
2. Ищет релевантные chunks в Qdrant
3. Формирует промпт для DeepSeek
4. Возвращает AI-ответ с evidence cards

## Часть 9. База данных — что где хранится

### Три базы

| БД | Владелец | Назначение |
|----|---------|-----------|

| `synapse` | Matrix homeserver | Внутренние данные Synapse |
| `mautrix_telegram` | Telegram bridge | Данные моста |

### Таблицы superchat_app (простыми словами)

| Таблица | Аналогия | Что хранит |
|---------|----------|------------|
| `pilot_invites` | Список гостей на вечеринку | Кто может войти в систему |
| `app_users` | Паспортный стол | Зарегистрированные пользователи |
| `magic_links` | Одноразовые пропуска | Временные ссылки для входа |
| `api_sessions` | Карточки доступа | Активные API-токены |
| `matrix_identities` | Связь паспорт ↔ служебное удостоверение | Связь пользователя и Matrix-аккаунта |

| `chunk_build_checkpoints` | Закладка builder'а | Прогресс chunking |

## Часть 11. Фоновые процессы (Background Services)

Все фоновые процессы запускаются **только в Web** хосте.

| Процесс | Что делает | Частота |
|----------|-----------|---------|
| `MatrixSyncBackgroundService` | Синхронизирует сообщения из Matrix | непрерывно |
| Chunk Builder (через pipeline) | Группирует сообщения в chunks | каждые 5 сек |
| Chunk Indexing (через pipeline) | Создаёт embeddings и загружает в Qdrant | каждые 10 сек |
| Meeting Projection (через pipeline) | Обновляет meetings из chunks | каждые 15 сек |
| Resolution (через pipeline) | AI-проверка статуса work items | по событию |

Pipeline реализован через **Rebus** — message bus, который использует PostgreSQL как транспорт (или InMemory для тестов).

## Часть 12. Полиморфные Work Items

Work items — это полиморфная система с тремя типами:

### Request (запрос)
Кто-то попросил вас что-то сделать.
- Статусы: `AwaitingResponse`, `Answered`, `Declined`, `Expired`
- Пример: "Пришли отчёт до пятницы"

### Event (событие/встреча)
Запланированное событие с датой.
- Статусы: `PendingConfirmation`, `Confirmed`, `Cancelled`, `Completed`
- Может содержать ссылку на Zoom/Google Meet
- Пример: "Созвон завтра в 14:00"

### ActionItem (задача)
Конкретное действие к выполнению.
- Статусы: `ToDo`, `InProgress`, `Done`, `Dismissed`
- Пример: "Нужно подготовить презентацию"

### Общие свойства всех work items

- `Priority` (Normal, High, Low)
- `ReviewState` (NeedsReview, Approved, Rejected) — AI-результаты требуют одобрения
- `ResolutionKind` — как был закрыт (AI, вручную)
- `Confidence` — уверенность AI в извлечении (0.0 - 1.0)

## Часть 14. Embedding Service (Python sidecar)

### Что это

Маленький Python-сервис, который преобразует текст в вектор (embedding).

### Зачем

Для семантического поиска. Мы не ищем по точному совпадению слов — мы ищем по **смыслу**.

### Как работает

```
Текст "Пришли отчёт завтра" → [0.123, -0.456, 0.789, ...] (1024 числа)
```

### Варианты backend

| Backend | Когда использовать |
|---------|-------------------|
| `LocalService` | Локальный Python сервис с BGE-M3 моделью |
| `YandexCloud` | Yandex AI Studio (text-search-doc/query) |
| `mock` | Для разработки без реальных embeddings |

## Часть 15. Text Enrichment Service

Ещё один Python-сервис, который **обогащает текст** перед extraction:

- Нормализует формат дат
- Очищает текст от мусора

Опциональный — можно отключить через `TextEnrichment__Enabled=false`.
